# Wiki Query

위키는 **그래프**다. 평면 문서가 아니라 `[[링크]]`로 연결된 노드망이다. query 스킬은 이 그래프 구조를 활용해 관련 페이지를 찾는다.

## 왜 그래프 탐색인가

시드 페이지만 읽으면 핵심 연결을 놓친다. 예를 들어 "디지털 트윈과 온톨로지"를 물으면 키워드로는 `digital-twin.md`, `knowledge-graph.md` 2개만 히트하지만, 두 개념을 실제로 연결하는 답은 `syntheses/manufacturing-llm-integration.md`에 있을 수 있다. 이 syntheses는 시드 페이지가 `[[링크]]`로 참조하거나, 시드 페이지를 `[[참조]]`하는 백링크로만 발견된다.

## 워크플로우

### 1. 시드 수집 — 인덱스와 Grep **둘 다** 수행한다

인덱스와 Grep은 **대체 관계가 아니라 보완 관계**다. 찾는 것이 다르다:

- **인덱스 스캔**: `wiki/index.md`를 읽는다 → 카테고리별 제목·한 줄 설명 기반. **제목에 키워드가 없는 페이지는 놓친다.**
- **Grep 전문 검색**: `wiki/` 내 질문 핵심 키워드 리터럴 검색 → **본문에서만 언급된 동의어·유사 표현을 잡는다.**

#### 왜 둘 다 필요한가 (구체 예시)

질문: "디지털 트윈과 온톨로지 관련?"
- 인덱스만 돌리면: `digital-twin.md`, `knowledge-graph.md` 히트 → 여기서 멈추면 핵심 답을 놓친다
- Grep "온톨로지" 추가: `manufacturing-process-control.md`의 "FDC의 '왜?' → 온톨로지 인과추론" 섹션 발견 (이 페이지 제목에는 "온톨로지"가 없음)

**인덱스 히트 수와 무관하게 Grep을 수행한다.** "인덱스에서 찾았으니 충분해 보임"은 신뢰할 수 있는 신호가 아니라 검증되지 않은 추측이다.

#### 1단계 완료 체크리스트

아래 **두 산출물이 모두 손에 있어야** 2단계로 넘어간다:

- [ ] 인덱스에서 선택한 페이지 목록 (0개여도 명시)
- [ ] Grep 결과 목록 — 사용한 검색어도 기록

하나라도 비어 있거나 "할 필요 없음"으로 판단해 건너뛰었다면, 지금 즉시 돌아가 수행한다. **Grep 호출이 트랜스크립트에 없다면 이 단계는 미완료다.**

시드는 보통 2~4개. 너무 많으면 3단계에서 우선순위로 제한.

### 2. 그래프 1-hop 확장 — Outgoing과 Backlinks **둘 다** 수행한다

시드 페이지만 읽으면 **시드가 참조하지 않는 관련 페이지**를 영구히 놓친다. 확장은 두 방향이 다르다:

- **Outgoing links**: 시드 본문의 `[[wiki-link]]` 추출 → 시드가 **직접 연결한** 페이지
- **Backlinks**: Grep으로 시드 이름 역검색 (`grep -rn "\[\[concepts/digital-twin" wiki/`) → 시드를 **참조하는** 페이지

두 방향은 비대칭이다. Outgoing에 없는 backlink가 흔하다 — 예를 들어 `syntheses/manufacturing-llm-integration`은 `digital-twin`을 참조하지만 digital-twin.md는 이 synthesis를 참조하지 않는다. Backlinks를 안 하면 이런 페이지를 놓친다.

**Syntheses 우선**: 확장 후보 중 `syntheses/` 페이지는 우선순위를 높인다. "여러 페이지를 교차 분석한 통찰"이므로 관계형 질문("X와 Y는?", "비교해줘")의 정답을 담고 있을 확률이 높다.

#### 2단계 완료 체크리스트

아래 **두 산출물이 모두 손에 있어야** 3단계로 넘어간다:

- [ ] 시드들의 Outgoing `[[links]]` 목록
- [ ] 시드들로의 Backlinks 목록 (Grep 역검색 결과)

Backlinks Grep 호출이 트랜스크립트에 없다면 이 단계는 미완료다.

### 3. 읽을 페이지 선택 (컨텍스트 보호)

**대부분 3\~7개로 충분하다. 상한은 15개.** 15개를 넘긴다면 질문이 너무 광범위하거나 시드 선택이 잘못된 것이므로 사용자에게 재질문하거나 질문을 좁힌다. "15개를 채워야 한다"는 의미가 아니라, **질문에 답할 수 있는 최소한**으로 읽는 것이 목표다.

우선순위 (위에서부터 채운다):

1. 질문 키워드가 제목·aliases와 직접 매칭되는 시드
2. 시드에서 1-hop으로 연결된 syntheses (관계형 질문에 핵심)
3. 시드에서 1-hop으로 연결된 concept/entity
4. summaries는 질문이 "이 소스에 뭐 있었어?" 스타일일 때만 읽음 (허브 페이지이므로 본문에 내용 적음)

#### 3단계 진입 전 자문

"내가 지금 페이지를 고르려는 이유가 **검색이 실제로 완결되었기 때문**인가, 아니면 **'찾았다'는 만족감 때문**인가?" 1단계와 2단계의 체크리스트를 다시 본다. 비어 있는 항목이 있으면 거기로 돌아간다. 만족 편향은 LLM의 구조적 경향이라 인지하지 않으면 반복된다.

### 4. 정보 수집 및 종합

각 페이지에서 질문에 답할 정보를 추출하고 **어느 페이지에서 왔는지** 기록한다. 페이지 간 정보를 교차 비교하여:

- 동일 사실 확인 → 신뢰도↑
- 상충 사실 → 두 출처 모두 제시

### 5. 답변 생성

답변 끝에 반드시 아래 두 블록을 포함한다. Obsidian 백링크 추적과 쿼리 감사(어떤 페이지가 실제로 참조되었는지)를 위해 필수다.

```markdown
{답변 본문 — 본문 안에서는 [[링크]]를 인라인으로 쓰지 않는다}

**참조한 위키 페이지:**
- [[페이지1]] — {이 페이지에서 가져온 핵심 정보 한 줄}
- [[페이지2]] — {이 페이지에서 가져온 핵심 정보 한 줄}

**역기록:** {새 통찰이 있으면 생성한 페이지명, 없으면 "없음 — 기존 정보의 종합이므로 역기록하지 않음"}
```

### 6. 역기록 (write-back)

여러 페이지의 정보를 교차 분석하여 **기존 위키 어디에도 없는 새로운 통찰**이 발생한 경우에만 역기록한다:

1. `wiki/syntheses/`에 새 페이지를 생성한다
2. frontmatter에 `generated-from: query`를 추가한다
3. `wiki/index.md`에 등록한다

기존 정보를 재구성한 답변은 역기록하지 않는다. 역기록 여부와 그 판단 근거는 답변 끝에 항상 명시한다.

### 7. 로그 기록

답변을 사용자에게 보여준 뒤 `.wiki-log.md`에 쿼리 기록을 남긴다. 답변 생성과 로그 기록은 하나의 연산이다 — 답변만 하고 로그를 빠뜨리면 연산이 미완료된 것이다.

```markdown
## [YYYY-MM-DD] query | {질문 한 줄 요약}
- 시드: {키워드로 찾은 페이지}
- 확장: {그래프 탐색으로 추가된 페이지}
- 역기록: {있으면 새 페이지명, 없으면 "없음"}
```

## 답변 원칙

- 위키에 있는 정보만으로 답한다. 외부 지식을 섞지 않는다.
- 위키에 정보가 없으면 "위키에 해당 정보가 없습니다"라고 답하고, 소스 수집을 제안한다.
- 모순된 정보가 있으면 두 출처를 모두 제시한다.
- 답변은 간결하고 구조적으로 작성한다.

## 질문 유형별 힌트

| 질문 유형 | 예시 | 탐색 전략 |
| --- | --- | --- |
| 단일 개념 조회 | "wiki-llm이 뭐야?" | 시드 1\~2개로 충분. 확장 생략 가능 |
| 관계 질의 | "X와 Y 관련 있나?" | syntheses 우선 탐색. 1-hop 확장 필수 |
| 비교 | "X vs Y" | 두 시드 + 각각의 1-hop. syntheses 우선 |
| 개괄 | "위키에 뭐 있어?" | index.md만 읽고 카테고리 요약 |
| 사례 탐색 | "\~사례 있어?" | entities 우선 + 해당 entity를 참조하는 syntheses |
